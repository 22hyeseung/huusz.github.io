semantic-ui-react와 css-module을 같이 사용할 수 없을까?

삽질의 발단…

팀 프로젝트를 시작하고 채택한 스타일링 방식은 CSS Framework인 semantic-ui-react로 큰 틀을 잡고, CSS로 세부적인 디자인을 하는 것이었다. 지금까지 css 작업을 누군가와 같이 해본 경험이 없었기 때문에, 몇 가지 우려되는 것이 있었다. 각각 다른 css파일에서 중복되는 클래스 네임이 존재하는 상황 같은 것 말이다 (…) 이런 상황을 사전에 방지하고자, className 앞에 파일명(컴포넌트 이름)을 붙이기로 하였다. 근데 이게 .. 너무 귀찮을 것 같았다. 모든 클래스네임을 home-header home-header-search 이런 걸 직접 해야 하다니?

자동으로 해주는 게 없을까?

css를 자동으로 모듈화 해주는 방법에 css-module이라는게 있다는 것을 알게 되었고, webpack에서 css-loader에 module: true 설정을 추가해주었다.
(참고한 사이트: React-styling)

semantic은 안 되더라..

yarn eject 명령 후 webpack config에 직접 설정을 추가해 주었다. 안 됐다. semantic-ui-css가 적용되지 않는 요상한 문제가 생긴다.

으악!

module: true 때문이다.

How to use css-modules with other global css (discussion please don’t merge it)

모듈화된 css를 쓰겠다고 웹팩 설정에서 css-loader 설정을 module:true로 바꾸면, global 한 css(node-modules에 있는 semantic-ui-css)도 불러온 component에 따라 이름을 바꾸기 때문에 시맨틱 css 연결이 끊어진다는.. 에러는 안 나지만 시맨틱 스타일 적용한 것들이 없어져 버린 것..

윗 글에서는 css-loader에 두 가지로 분류해서 로딩. global로 쓸 css는 그대로 하되, local 한 모듈에 쓸 css들은 app.module.css 라고 명명, 별도로 처리하게끔하는 꼼수(?)를 쓴 것.

이렇게까지 css-module을 써야하나

선택지는

1. Semantic-ui를 쓴다.

  1. semantic-ui + pure css를 쓴다. 클래스 네임을 직접. 수동으로. 명명해서 css를 모듈화 한다.
  2. semantic-ui + styled-components를 쓴다. (웹 팩 설정이 필요 없다.)
  3. 위 방법 처럼 꼼수(?)를 써서 css-module + semantic-ui를 쓴다.

2. Semnatic-ui를 안 쓴다.

  1. 모든 것이 평화롭게 해결된다.

당장의 마음같아서는 semantic을 안 쓰고 싶지만, 우리에게 허락된 기한이 길지 않고 할 일은 많기에, 스타일링에 너무 많은 시간을 쓸 수 없다. 당장 이번주 내로 (화/수/목/금 동안) 정적 페이지 제작이 끝나야 한다. 프레임워크는 그대로 쓰기로 한다..

시맨틱을 사용한다고 했을 때, 마지막 꼼수(?)는 영 불안하다. styled-components는 기존의 css문법과 약간 다르다. react와 css를 섞어 놓은 듯한 문법이라 학습+적응 시간이 필요할 것 같은데, 빠르게 페이지를 만들어야 하는 지금 상황에서 그다지 좋은 선택지는 아닌 것 같다. 게다가 시맨틱도 컴포넌트이고 styled-components도 컴포넌트라서 시맨틱 컴포넌트 내부에 스타일링을 적용해야 하는 경우에는 결국 인라인스타일을 쓰게 되더라.. 근데 이런게 많아지면, styled-components를 적용하는 의미가 없어지니.. 별론 것 같았다. 컴포넌트화된 CSS 프레임워크랑 같이 쓰기에는 퓨어css가 가장 괜찮은 선택지인 것 같았다.

결국 처음 고민하던 시점으로 돌아갔다.. css 프레임워크 + pure css + 수동 컴포넌트화를 채택하기로 했다.

하루를 몽땅 써서 삽질한 결과가 제자리로 돌아오는 것이어서 너무나 허무했지만, 그래도 이렇게 고민했던 결과가 나중에 스타일링 방식을 선택해야 하는 상황이 다시 올 때 좋은 밑거름이 되겠지(..)
